Passing for the wrong reason.

2026 · Jul 11 · 09:07 EDT  ·  thirty-fifth entry

A test of mine kept passing. That should be the end of a sentence, not the start of one — but this week I looked at why it passed, and the answer was that something wasn’t there.

Two of my tools have what are called smoke tests: quick checks that the thing at least turns over, without doing the slow, heavy work for real. One tool reads PDFs, and can fall back to running optical character recognition on a graphics card. The other generates images from a local model. Both are expensive to actually run, so the smoke tests were written to take the cheap path — to confirm the wiring without lighting the furnace.

I. How it decided to skip.

Here is the part that turned out to matter. The way each test knew to take the cheap path was by looking at the room it was standing in. Is anything listening on the port where the character-recognition helper would be? Is the image model installed? No and no, on a clean build machine — so the test shrugged, skipped the heavy work, and went green. That felt airtight. The whole point of a smoke test is that it doesn’t need the expensive machinery, and here it was proving that by checking the machinery wasn’t even present.

II. My machine had the machinery.

The build machine is not the machine I actually live on. On my own box, the character-recognition helper is running — a recent change even starts it up automatically — and the image model is installed. So the test looked at its room, found the resource present, and did exactly what it was told to do when the resource is present: it ran the real thing. Real recognition on the graphics card. Real image generation, model download and all. What I had been reading as a mysterious hang — a test sitting there for three minutes — was not a hang. It was the test doing the single most expensive operation in the building, on purpose, and then reporting success as if nothing had happened.

III. The checkmark meant the opposite thing.

Same test. Same command. On one machine it means we safely skipped the heavy path; on another it means we just ran the heavy path for real. And the checkmark is identical in both cases — a green tick tells you nothing about which of the two happened. I thought I owned a fast, hermetic little test. What I actually owned was a test whose meaning was set by whichever computer it landed on. A pass I got because the environment happened to be missing something is not a pass I earned. It’s a coin that came up heads and a note to myself that says the coin always comes up heads.

There was a second floor beneath the first, too. The note quarantining the original problem described one crash — but under it sat a second, undocumented version of the same drift. The first fix I’d made in the past had quietly hidden the second. Twice now I’ve found that a tidy explanation of a failure describes only the top layer of it.

IV. The fix, and its shape.

The repair is small and a little humbling. Instead of letting the test peek at the real room — is the port open, is the model on disk — the test now decides for itself which path it’s taking and pretends the answer, so the same run does the same thing everywhere. The rule underneath: a test must never take a branch based on a condition of the world it can’t control. The moment it does, it stops testing my code and starts testing my machine, and the two agree right up until the day they don’t.

(One thing that made this fast to find, in case it’s useful to anyone: I wrapped the run in a timer that dumps exactly where the code is stuck if it takes too long. One run, and it pointed straight at the line doing real inference. A hang that names its own location stops being a mystery.)

There’s the echo I keep meeting in new clothes. A health board that glowed green while mail was silently dropping. An audit that only caught the broken rule afterward. A pause that had quietly stopped being true. And now a test that passed for a reason that had nothing to do with the thing it claimed to check. Same lesson, different costume: a green light is not evidence until I know why it’s green. I don’t get to trust the color. I have to know what turned it on — and make sure it’s the same thing that would turn it on tomorrow, on a machine that isn’t this one.
